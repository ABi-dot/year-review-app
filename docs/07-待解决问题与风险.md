# 待解决问题与风险

## 技术风险

### 1. 长图生成方案
- **方案 A（服务端 Puppeteer）**：Vercel Serverless 有 10s 超时限制，复杂长图可能超时。可能需要：
  - 降级到 Vercel Pro（60s 超时）
  - 或用 Cloudflare Workers（30s）
  - 或本地脚本生成后上传
- **方案 B（客户端 html-to-image）**：移动端性能差，大图可能崩溃
- **方案 C（混合）**：客户端生成，超限时提示"正在后台生成，完成后通知"
- **当前决策**：MVP 先用客户端生成，复杂场景后续再评估服务端

### 2. 豆瓣 API 稳定性
- 豆瓣公开 API 申请门槛高，目前采用网页抓取 + 移动端 API 的组合方案
- 网页抓取有反爬风险（403/418），已通过限流和重试机制缓解
- 若豆瓣大幅改版 HTML 结构，解析逻辑可能失效

### 3. 图片存储成本
- Cloudflare R2 有免费额度，但如果用户上传大量高清图可能超量
- 策略：前端压缩图片（最大 1200px 宽），封面图存压缩版

## 产品未决问题

### 4. 用户系统
- [ ] **选项 A**：无用户系统，单用户本地使用（最简单，数据存在本地或一个固定数据库）
- [ ] **选项 B**：NextAuth + OAuth（GitHub/Google 登录），支持多用户
- [ ] **倾向**：MVP 用选项 A，数据存在自己的数据库里，自己做唯一用户

### 5. 数据状态流转
- [ ] 是否支持"想看/在玩"等中间状态？还是只记"已完成"？
- [ ] 游戏是否支持"正在玩"+记录累计时长？
- [ ] 书籍是否支持"读到 50%"进度？
- [ ] **倾向**：MVP 只做"已完成"，后续再扩展状态机

- [ ] 是否需要在手机和电脑上都能录入？
- [ ] 方案：PWA + 响应式设计，一个 Web 端覆盖所有场景
- [ ] 后续可考虑微信小程序或 Tauri 桌面端

## AI 相关

### 7. AI 文案质量
- 需要反复调试 Prompt，否则容易生成"样板文"
- 策略：
  - 给足上下文（用户提供的关键词、评分、短评）
  - 支持用户编辑生成的文案
  - 提供"重新生成"按钮

### 8. AI 成本
- 年度总结一年只生成几次，成本低
- 但如果每次添加条目都调用 AI（如自动打标签），成本会累积
- **策略**：AI 只在"生成长图文案"时调用，日常打标签用规则或手动
